   * right now gap lines are just trailing whitespace owned by the preceeding parsed node/block. but in an outliner there is a difference between the trailing space that is part of a block (in text it would normally be actual indentation whitespace on a separate line, part of the block at the same indentation level), and the empty lines with only a newline character between the blocks (that normally isn't part of the tree).
   * so when I see two sibling nodes and place cursor at the beginning of the second node, and press backspace, I expect that node to get merged with the first one and the text to be appended to the end of it. but because of the gap lines, it doesn't happen like that, nodes get merged, but the text gets appended in place of that gap line. or the gap line gets deleted (e.g. between a heading and a paragraph), but the nodes are not merged. etc.
   * my intuition is that we need to skip over the gap lines and treat them as if they weren't part of the text. they are an implementation detail of the markdown representation which requires or allows to separate blocks with empty lines, but in an outline tree structure, they are not structurally functional and the ownership of those lines is not clear to the user.
   * think about it and let's decide how to deal with this.
* lists are not handled well: if I place cursor at the beginning of any list item and press backspace, I expect it to get merged with the previous list item or its parent paragraph. but instead, it deletes the space after the `-` dash-bullet. turning it into a strange paragraph.

* if a list item has a child and I decide to split the parent node by pressing enter in the middle of the text, I expect that to create a new first child (new preceeding sibling to the existing child). but right now it creates a sibling of the parent and it feels very unnatural because it jumps over the child.
   * if the node doesn't have a child, splitting it should create a sibling.

ok. great. here are some more observations:

* one thing that is the consequence of the adjustment I requested before is that splitting or adding a new node after one with children introduces a new child. in some cases this introduces a "soft-lock".
   * say we have 2 paragraphs: A (simple) and B with children (a list). then we merge A and B, B becomes part of A, its children are now owned by the merged paragraph node. if I press enter after that, B becomes A's new first list-child.
      * first, this might seem a bit unintuitive that pressing backspace, then enter doesn't return to the same initial state. I'm not saying it has to be like that, but it's just an observation that is worth pointing out.
         * I've thought about treating paragraphs and lists a bit differently, so thay pressing enter in a paragraph introduces always a new paragraph (new or split), even if there is a list after it, it will become the subtree of the new paragraph. this feels natural in theory, but not very consistent if we keep the rule that in lists the same action would introduce a new child. so I think a better solution is to keep it consistent, but allow outdenting in place (next point)
      * second, after these motions, it's impossible to return to the original state with the structural operations: after the new split, if I'm on B (now list-child of A), and press shift-tab, it becomes A's sibling, but jumps across the list. A stole B's children!
         * I don't think this is correct behavior. it's similar to the correction we made before. I think on shift-tab, B should become a paragraph with its original children, i.e. outdent in-place, instead of becoming A's _new_ sibling. 
            * this is consistent with Logseq: if a sibling can be outdented in place, it happens so and its following ex-siblings become its children. note indenting after that is not symmetric: now that node has children and they will be indented too.
