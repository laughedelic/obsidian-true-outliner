## 1. What parser does Obsidian actually use (2026)?

**No confirmed answer — Obsidian's internal parser is closed-source and undocumented, and this hasn't changed.** There is no official spec or source-visible implementation.

- Obsidian dev team confirmed in 2021/2022 there's no reference spec: "There is no reference spec and is not currently on the roadmap" (WhiteNoise, Obsidian staff, Oct 2022), and "we try to use commonmark with some minor additions" (WhiteNoise, Dec 2021) — [Is there a parser/renderer reference spec?](https://forum.obsidian.md/t/is-there-a-parser-renderer-reference-spec/29504)
- The "remark-parse 8 or earlier with commonmark option" claim is **not from 2024** — it's from a forum post dated **April 11, 2026** (user ile-24556), explicitly labeled speculation based on observed edge-case similarities to Prettier/remark-parse-8, "unconfirmed by Obsidian developers" — [Obsidian Markdown Parser](https://forum.obsidian.md/t/obsidian-markdown-parser/55922). A separate 2026-01-26 comment on the same thread calls the parser a "black box," complaining devs are "forced to add remark as a dependency" themselves rather than being given access to Obsidian's own parser.
- For the **editor (Live Preview/CM6)**: Obsidian's editor runs on CodeMirror 6, whose stock markdown mode uses the Lezer parser (`@lezer/markdown`, via `@codemirror/lang-markdown`). However, the community project `lezer-markdown-obsidian` (last pushed 2023-12-27) states explicitly: **"This is not the parser that Obsidian itself uses, and all parsing is a best attempt to match the way Obsidian parses markdown"** — [erykwalder/lezer-markdown-obsidian](https://github.com/erykwalder/lezer-markdown-obsidian). This strongly implies Obsidian ships a **proprietary/customized tokenizer** for Live Preview rather than the stock open-source Lezer markdown grammar.
- For **reading-mode rendering**: no verifiable public statement either way; community analyses variably guess remark-based or markdown-it-influenced behavior (e.g., one third-party doc attributes an HTML-block-parsing quirk to "the underlying CommonMark specification and the Markdown-it parser," but this is inference, not an Obsidian source statement).
- **No public API for a parsed AST exists, confirmed as still true.** `MarkdownRenderer.render()` produces HTML only, with `renderMarkdown` marked deprecated in typings ([obsidian-api](https://github.com/obsidianmd/obsidian-api), [docs.obsidian.md/MarkdownRenderer](https://docs.obsidian.md/Reference/TypeScript+API/MarkdownRenderer)). Forum thread ["Access parsed Markdown"](https://forum.obsidian.md/t/access-parsed-markdown/25398) (2021, no Obsidian staff reply) shows the standard workaround is plugin authors running their own `mdast-util-from-markdown` pass on the raw file text. This pattern is unchanged in 2026 per the newer forum thread above.

**Confidence: high** that no public AST API exists and the internal engine is undocumented; **low-to-moderate** on any specific claim about which library (if any pre-existing one) Obsidian's core is actually built on — treat all such claims, including the remark-8 one, as community speculation.

## 3. Round-trip / stringify fidelity

**mdast-util-to-markdown (remark-stringify) is explicitly not byte-preserving and not fully structure-preserving.** This is a known, acknowledged limitation, not a niche bug:

- Confirmed open issue: [`mdast-util-to-markdown` #12](https://github.com/syntax-tree/mdast-util-to-markdown/issues/12) — parsing `***emphasis*in emphasis*` and stringifying it back produces `\***emphasis*in emphasis*`, which **re-parses into a structurally different AST** (nested emphasis collapses to flat). Labeled "confirmed," still open as of my check (repo last pushed 2026-04-24, 8 open issues total), no maintainer resolution stating round-tripping is a design goal.
- General project stance (from repo docs/README and community discussion threads like [remark-stringify formatting questions #1166](https://github.com/orgs/remarkjs/discussions/1166)): the stringifier "does its best to serialize markdown to match the syntax tree," but complete round-tripping is stated to be impossible in general since arbitrary values can be injected into the tree, and there are known issues around escaping and spacing preferences that aren't fully configurable.
- **Practical implication**: for a tool that must preserve exact source formatting (whitespace, list marker style `-` vs `*`, blank-line patterns, indentation) across incremental edits, remark-stringify alone is not safe to trust for un-touched parts of the document.

**On using position offsets to slice the original source instead of stringifying:** unist `Position`/`Point` nodes do carry 0-indexed `offset` fields alongside line/column, and `mdast-util-from-markdown` populates these reliably for CommonMark-compliant parses (I did not find any specific report of the offsets being unreliable or missing for standard nodes). This is a well-established pattern (also used by tools like Prettier's markdown printer and various codemods) — slice the raw source with node offsets for content you didn't change, and only synthesize new markdown for nodes you actually created/modified. I did not find a single dedicated package/utility explicitly branded for this "surgical slice" pattern, but the primitives (`unist-util-position`, position offsets themselves) support it. **Confidence: moderate** — this is inferred from unist's documented data model plus known practice in the ecosystem (e.g., Prettier), not from an explicit case study I found and verified.

## 4. micromark as a standalone tokenizer/boundary detector

**Yes, this usage pattern is supported by design and has prior art**, though I found less explicit "here's a project doing exactly that" documentation than expected.

- micromark is explicitly architected as a layered system: the core tokenizer (`micromark-core-commonmark` + `micromark`) emits a flat event stream (start/end events referencing token spans with position/offset info) that is **decoupled from any specific output**. The micromark README states the parsing layer "can still be used separately to build ASTs, CSTs, or many other output formats" beyond its own HTML compiler — [micromark/micromark](https://github.com/micromark/micromark) (repo pushed 2025-05-10, 2,196★, actively maintained, 15 open issues).
- `mdast-util-from-markdown` itself is the reference example of consuming micromark's event stream and building a different tree shape (mdast) on top — i.e., "use micromark for tokenization, build your own tree" is literally how remark itself is built, which is the strongest piece of prior art: the ecosystem's own two-layer design (micromark = tokenizer, mdast-util-from-markdown = tree builder) is exactly the "boundary detector + keep your own tree" pattern you're asking about.
- I did not find a clearly documented third-party project that uses bare micromark (without any `mdast-util-*` tree builder) purely as a boundary/offset oracle while feeding a fully custom, non-mdast data structure. This may well exist (e.g., inside MDX tooling, Prettier's markdown plugin, or Astro's content pipeline) but I could not confirm it directly — **flag as unverified**, worth a targeted look at Prettier's `markdown` printer source or Astro's content-collections markdown handling if this pattern matters to the decision.
